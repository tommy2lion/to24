一、实现 person_like 算法：基于规则的启发式求解
你的规则分类非常清晰，几乎涵盖了人类解24点的主要策略。将这些规则转化为代码，可以设计一个规则引擎，按优先级依次尝试。

规则优先级设计（从易到难）
特解规则：某些经典难题（如 [5,5,5,1] → (5-1/5)*5）可以直接查表，但为了体现“人脑”而非记忆，可以只作为最后手段。不过，既然是模拟人类，可以保留一些常见特解。

简单连加/连乘：如果四个数可以直接相加或相乘得到24，直接返回。

因子匹配：这是你列出的核心规则。例如：

有一个4，看剩下三个数能否算出6。

有一个6，看剩下三个数能否算出4。

有一个8，看剩下三个数能否算出3。

有一个3，看剩下三个数能否算出8。

以及更广泛的：2→12, 8→16, 6→18, 4→28, 8→32, 3→21 等。

组合匹配：将四个数分成两组，每组运算后组合，如 (a op b) op (c op d) 或 (a op b) op c) op d 等，但用启发式剪枝。

回退到简单穷举：如果所有规则都失败，则调用 simple 算法作为保底（因为人类最后也会尝试穷举）。

数据结构设计
python
class PersonLike:
    def __init__(self):
        self.rules = [
            self.rule_special,          # 特解
            self.rule_all_add,           # 全加
            self.rule_all_mul,           # 全乘
            self.rule_factor,             # 因子匹配 (核心)
            self.rule_group,               # 分组匹配
        ]
    
    def solve(self, a, b, c, d, return_expr=False):
        nums = [a, b, c, d]
        # 尝试每条规则，直到成功
        for rule in self.rules:
            result = rule(nums)
            if result:
                if return_expr:
                    return True, result
                return True
        # 最后调用simple
        return self.simple_fallback(nums, return_expr)
核心规则实现示例
以“有一个4，剩下三个数算6”为例：

def rule_factor(self, nums):
    # 遍历每个数作为“锚点”
    for i, anchor in enumerate(nums):
        others = nums[:i] + nums[i+1:]
        target = 24 / anchor if anchor != 0 else None
        # 检查整数目标（如 anchor=4 → target=6）
        if target == int(target):
            # 尝试用三个数算出 target
            expr = self.can_make_target(others, int(target))
            if expr:
                return f"({expr}) * {anchor}"  # 或者 / 根据运算
    return None
can_make_target 可以是一个递归函数，尝试用三个数通过加减乘除得到目标值，同样可以应用小规模穷举。

关于“活性数”的扩展
你的观察很准：3、4、6、8 确实是最常见的因子。可以构建一个映射表，比如 {3: [8, 21, 27, 72], 4: [6, 20, 28, 96], 6: [4, 18, 30, 144], 8: [3, 16, 32, 192]}，但注意有些目标可能超过24（如 4*24=96），需要组合成 24 = anchor * target 或 anchor + target 等。

系统架构
统一算法接口

比赛逻辑

从一副扑克牌（1-10，每个数字最多4张）中随机抽取4张牌作为题目。

每个算法依次尝试求解，记录结果（正确/错误）和时间。

可以设计积分制：答对得1分，超时或答错扣分。

可设置回合数，最终决出胜者。

Web 界面

前端展示题目、算法实时状态、得分榜。

用 Koa 提供 REST API：POST /match 开始比赛，GET /status 获取实时结果。

可以使用 Server-Sent Events 或 WebSocket 推送进度。

防止作弊（代码行数限制）
可以在提交算法时统计代码行数（去除空行和注释），超过限制则拒绝参赛。或者更严格地，在沙箱中运行并监控执行时间，防止查表（如果算法直接返回预存结果，时间极短，可以检测）。

技术实现要点
动态加载算法：参赛者将代码上传或粘贴，后端保存为临时文件，通过 importlib 动态加载。

沙箱安全：如果允许多人提交，需要考虑安全性，可以在 Docker 容器中运行，限制 CPU 和内存。

计时：使用 time 模块测量 solve 执行时间，超过阈值（如1秒）判为超时。

关键疑问：

出牌顺序：是双方同时出牌（即都不知道对方出的牌），还是轮流出一方先出？如果同时出，算法需要预测对方可能出的牌？这会更复杂，但更真实。

二、策略博弈的引入
这个规则下，算法不仅需要求解能力，还需要出牌策略：

如果手牌中有强组合（容易解出），可能会想保留，而用弱牌去试探。

也可能故意出难牌，希望对方无解，从而自己收牌。

甚至可能通过观察对方已出的牌，推断对方手牌分布（如果算法有记忆）。

这其实是一个不完全信息博弈，类似于扑克游戏。设计这样的 AI 会非常有趣，也是你项目的一大亮点。
